请基于项目代码、目录结构、README、配置文件、测试脚本和样例数据，对项目进行工程化代码评审。

## 评审目标
1. 检查项目结构清晰度与模块职责合理性；
2. 验证课程核心功能链路是否有代码证据；
3. 检查数据从采集→清洗→解析→知识库→问答→Benchmark 的贯通性；
4. 检查字段、接口、配置、测试的一致性；
5. 检查业务逻辑实现质量（流程完整性、业务规则正确性、边界条件处理、模块间数据契约）；
6. 给出可执行的修改建议。

## 重要原则
1. 不要求目录名、接口名、函数名与参考完全一致，等价实现且职责清楚即可；
2. 没有代码/配置/数据/测试/日志/README 证据的功能，不应默认完成；
3. 评审聚焦工程结构、模块边界、数据贯通、可运行性和业务逻辑实现质量（流程正确性、规则落实、边界覆盖）；不评价界面美观，不做安全专项，不逐项检查日志格式、类型标注、代码风格等辅助性规范；
4. 对学生自定义实现给出改进空间，不因命名不同直接判为不符合；
5. **前端代码不纳入详细检查**——前端目录是否存在、是否能与后端对接即可，不对前端代码质量、框架选型、UI 细节做逐项评审。

## 三、核心功能模块检查

### 1. 数据采集模块
检查是否具备：
- 公开网页/文档数据源采集
- 列表页→详情页数据入口处理
- 标题、正文、发布时间、来源地址等元数据提取
- 附件识别与附件信息记录（附件下载非强制，但需记录附件元数据）
- `document_id`（文档唯一编号）和 `attachment_id`（附件唯一编号）的生成
- 分页、URL 去重（含 `normalized_url`）、请求重试、空内容等基础处理
- 内容去重（`content_hash` 内容指纹）
- 原始采集结果保存到 JSON/JSONL/数据库

### 2. 数据存储模块
检查是否具备：
- 结构化元数据存储（MySQL/SQLite/PostgreSQL 等），至少包含 `web_page` 和 `attachment` 两张核心表
- 原始文件存储（S3/MinIO/本地对象目录等），HTML 和附件分目录保存
- 元数据与文件之间的 `object_key` 关联字段
- 附件通过 `document_id` 关联所属网页
- 后续模块能读取这些数据

> 课程推荐 MySQL + S3/MinIO，等价实现需满足"结构化元数据管理"和"文件本体可追溯保存"。

### 3. PySpark 数据处理模块
检查是否具备：
- **真实使用 PySpark**（`SparkSession`、DataFrame API），非仅在文档中提及
- 读取采集结果（JSONL）或数据库导出数据
- 去重（`document_id` 去重 + `content_hash` 内容去重）、空值处理、字段标准化、时间格式转换
- 生成 `data_status` 质量标记字段（`valid`/`empty_content`/`short_content`/`invalid_time`）
- 清洗结果保存为 Parquet
- 下游文档解析只处理 `data_status="valid"` 或经确认可用的数据

### 4. Spark SQL 统计模块
检查是否具备：
- Spark SQL 或 DataFrame 查询完成统计分析（栏目分布、时间分布、附件类型、数据质量）
- 统计结果保存或可被后续系统读取

> **关键约束**：Spark SQL 是**离线**统计，Agent 的 `statistics_query` 读取预计算结果或数据库查询，**不得在用户每次提问时实时启动 Spark 作业**。检查代码中是否存在每次调用 `/api/qa` 时执行 `spark.sql()` 的情况。

### 5. 文档解析模块
检查是否具备：
- HTML 网页正文解析
- PDF、Word（DOCX）、Excel（XLSX）等至少若干类附件解析
- 解析结果统一为文本单元（`units` 结构），保留 `document_id`、`attachment_id`、文件名、来源地址、页码、工作表名
- 对解析失败、空文本、过短文本有基本处理
- 说明已支持格式和未支持格式

### 6. 知识库构建模块
检查是否具备：
- 文本清洗（去 HTML 标签、合并空白、清除乱码）
- 文本分块（按段落+句子切分，控制块长度 500 字符左右，相邻块 30-50 字符重叠）
- `chunk_id`（文本块唯一编号，推荐格式 `{document_id}_chunk_{index:04d}`）
- 文本块关联 `document_id`、`attachment_id`、`source_url`、`file_name`、`object_key`、`page_number`、`sheet_name`
- 向量化代码（课程推荐 `BAAI/bge-m3`）
- **向量归一化**（`normalize_embeddings=True`）——`IndexFlatIP` 内积检索的前提
- FAISS `IndexFlatIP` 索引构建
- 向量编号与 `chunk_id` 映射保存
- 索引文件和映射文件持久化

> 使用其他向量模型/向量库需在 README 或代码中说明原因，并验证检索和来源追溯能力。

### 7. RAG 问答模块
检查是否具备：
- 用户问题向量化（与知识库同模型、同归一化）
- FAISS Top-K 检索（推荐先召回 10 候选，过滤后保留 5）
- 检索结果过滤（过短文本、低相似度）+ 去重
- 检索内容组织进提示词（使用 `[资料1]`、`[资料2]` 编号，来源含文件名和页码）
- 调用大语言模型（DeepSeek，通过 LangChain `ChatDeepSeek`，`temperature` 0.1-0.3）
- 回答结果提取引用编号并映射回真实来源
- 来源信息返回（必须可追溯到检索结果或元数据）
- 无结果/无知识时的处理

### 8. Agent 与工具调用模块
检查是否具备：
- 对不同问题类型进行任务判断或路由
- 实现以下**五类工具中的至少四类**：

| 工具 | 功能 |
|------|------|
| `knowledge_search` | FAISS 知识检索 |
| `page_query` | 网页标题/时间/来源查询 |
| `statistics_query` | 数量/分类/时间统计 |
| `attachment_search` | 附件查询 |
| `generate_download_url` | 附件临时下载链接生成 |

- 工具参数校验
- 统一返回格式 `{"success": bool, "data": any, "message": str}`
- 工具边界清晰，非所有逻辑写在一个超大判断函数中
- 数据库查询由预编写方法完成，**不允许 Agent 直接生成 SQL**
- 能将工具结果整合为用户回答

> 工具名称可不同，请建立"学生实现名称 → 课程推荐能力"的映射。

### 9. 后端服务模块
检查是否具备：
- 问答接口（推荐 `POST /api/qa`）
- 附件访问/下载接口（推荐 `GET /api/attachments/{attachment_id}/download`）
- 请求参数校验（`question` 非空、长度限制等）
- 统一响应结构（`status`/`question`/`answer`/`sources`/`attachments`/`session_id`）
- 错误处理与日志记录
- 与 RAG、Agent、知识库、配置模块的合理调用关系

> 课程推荐 FastAPI + LangChain `ChatDeepSeek`。接口路径和字段名可不同，需说明等价关系。

### 10. 前端模块（简要检查）
仅确认前端目录存在且能调用后端接口即可，不对前端代码做详细评审。

### 11. Benchmark 与测试模块
检查是否具备：
- Benchmark 输入读取（JSONL，含 `id` 和 `question`）
- 批量调用问答接口
- **每条测试使用独立 `session_id`**（如 `benchmark_test_001`），不共享聊天历史
- 保留原始 `id` 和 `question`
- 输出 `answer`、`sources`（完整网址数组）、`attachments`（文件名数组）
- 失败样例保留记录（空答案 + 空数组），不跳过
- 结果保存为 JSONL

> 字段名可不同，需能清晰映射到 `id`/`question`/`answer`/`sources`/`attachments`。

## 四、关键数据贯通检查

逐项检查以下字段是否贯穿上下游（字段名可不同，语义须稳定）：

| 推荐字段 | 应贯穿的阶段 |
|----------|-------------|
| `document_id` | 采集结果 → 数据库 → 清洗结果 → 解析结果 → 文本块 → 检索结果 → 来源展示 |
| `attachment_id` | 附件记录 → 附件解析 → 附件查询 → 附件下载 |
| `chunk_id` | 文本分块 → 向量映射 → FAISS 检索结果 → RAG 来源 |
| `source_url` | 采集阶段 → 检索结果 → 问答来源 → Benchmark 输出 |
| `object_key` | S3 上传 → 数据库记录 → 文档解析（读取 S3）→ 下载链接生成 |
| `session_id` | 前端会话 → 后端请求 → Benchmark 独立测试 |

输出要点：贯通良好的字段 / 某阶段丢失的字段 / 被重复生成的字段 / 命名不一致但语义等价的字段映射 / 导致来源或附件无法追溯的字段。

## 五、接口与数据格式检查

采用"推荐格式 + 等价接受"方式检查。

**问答请求**推荐字段：`session_id`、`question`、`use_agent`
```json
{"session_id": "session_001", "question": "申请论文答辩需要提交哪些材料？", "use_agent": true}
```

**问答响应**推荐字段：`status`、`question`、`answer`、`sources`、`attachments`、`session_id`
```json
{
  "status": "success",
  "question": "...",
  "answer": "...[资料1]",
  "sources": [
    {
      "source_no": 1,
      "document_id": "doc_0001",
      "file_name": "研究生论文答辩管理规定.pdf",
      "source_url": "https://example.edu.cn/info/1234.htm",
      "page_number": 3,
      "sheet_name": null,
      "content_preview": "..."
    }
  ],
  "attachments": [
    {
      "attachment_id": "att_0001",
      "file_name": "论文答辩申请表.docx",
      "file_type": "docx"
    }
  ],
  "session_id": "session_001"
}
```

**工具结果**推荐格式：`{"success": true, "data": {}, "message": "执行成功"}`

**Benchmark 输出**推荐 JSONL，每行含 `id`、`question`、`answer`、`sources`（完整网址）、`attachments`（文件名）
```json
{"id": "test_001", "question": "...", "answer": "...", "sources": ["https://..."], "attachments": ["申请表.docx"]}
```

字段名不同时给出映射表。

## 六、业务逻辑实现检查（5 项）

聚焦数据处理流程的业务逻辑正确性，逐模块检查以下维度。每项给出「符合 / 部分符合 / 不符合」判定，并附代表性代码位置。

### 6.1 业务流程完整性

检查端到端数据处理链路是否可贯通执行，各阶段输入输出是否衔接：

| 检查项 | 说明 |
|--------|------|
| 采集→存储 | 采集结果是否可直接被存储模块消费（字段结构一致，非人工中转） |
| 存储→清洗 | PySpark 清洗脚本是否从数据库或 JSONL 读取采集结果，非手动导出 CSV 喂入 |
| 清洗→解析 | 文档解析是否只处理 `data_status="valid"` 的数据；解析模块是否能读取 S3/本地文件 |
| 解析→知识库 | 解析产出的 `units` 结构是否可直接输入分块与向量化流程 |
| 知识库→问答 | FAISS 索引与映射文件是否被 RAG 检索模块加载并正常使用 |
| 问答→Benchmark | Benchmark 脚本是否调用 `/api/qa` 接口获取结果，非直接调用内部函数绕过服务层 |

**判定标准**：
- **符合**：上述 6 条链路均可在代码中找到衔接证据（import 或函数调用关系清晰）
- **部分符合**：存在 1-2 处断层需人工介入（如手动复制文件、手动改路径）
- **不符合**：3 处以上断层，各模块独立运行但未串联为整体流程

### 6.2 业务规则正确性

对照三中各模块的功能要求，检查代码中的业务决策逻辑是否落实到位（具体参数规格参见三，此处仅检查"是否实现且方向正确"）：

| 检查项 | 对应三中的要求 | 检查要点 |
|--------|---------------|---------|
| 去重规则 | 三.1 / 三.3 | 是否同时存在 URL 去重和内容去重；`content_hash` 是否基于正文计算 |
| 质量标记规则 | 三.3 | `data_status` 四类枚举值的判断分支是否完整；阈值设置是否合理 |
| 分块规则 | 三.6 | 块长度和重叠量是否在合理区间；`chunk_id` 格式是否可追溯到源文档 |
| 向量检索规则 | 三.6 / 三.7 | 索引类型与归一化是否匹配（`IndexFlatIP` ↔ `normalize_embeddings=True`） |
| Agent 路由规则 | 三.8 | 是否存在问题类型判断逻辑（知识问答 / 网页查询 / 统计查询 / 附件查询），各分支是否可达 |
| Benchmark 独立会话 | 三.11 | 每条测试是否使用独立 `session_id`（如 `benchmark_test_001`），非共享聊天历史 |

**判定标准**：
- **符合**：上述 6 条规则均在代码中有对应实现，逻辑方向正确
- **部分符合**：1-2 条规则未实现或实现有偏差
- **不符合**：3 条以上关键规则缺失或逻辑明显错误

### 6.3 边界条件与异常路径

检查业务异常场景是否有代码层面的处理逻辑（三中已列出各模块需处理的基本异常场景，此处检查代码是否真正落实）：

| 检查项 | 对应三中的要求 | 检查要点 |
|--------|---------------|---------|
| 空内容处理 | 三.1 / 三.3 | 空正文是否标记而非丢弃；下游是否跳过空内容 |
| 解析失败处理 | 三.5 | 单文件解析失败是否阻断整批；失败原因是否可追溯 |
| 检索无结果处理 | 三.7 | 无结果时是返回"未找到"还是报错/编造 |
| 附件缺失处理 | 三.9 / 三.2 | S3 文件不存在时返回明确错误还是 500 |
| 超长输入处理 | 三.7 | 上下文超出 token 限制时是否有截断 |
| Benchmark 失败保留 | 三.11 | 失败样例是否保留空记录而非跳过 |

**判定标准**：
- **符合**：至少 4 种场景有明确的代码处理逻辑
- **部分符合**：2-3 种有处理
- **不符合**：仅 0-1 种有处理，大量边界无防御

### 6.4 模块间数据契约

检查模块间传递的数据结构是否一致，避免字段名随意变化导致下游解析失败（关键字段定义参见四，接口格式参见五）：

| 检查项 | 检查要点 |
|--------|---------|
| 采集→存储→清洗 | 采集产出的 JSON/JSONL 字段是否稳定；Parquet schema 与下游读取代码是否匹配 |
| 解析→知识库 | `units` 结构是否包含来源追溯所需字段（`document_id`、`attachment_id`、`source_url` 等） |
| 分块→检索→问答 | 文本块 JSONL 是否包含 `chunk_id` 及溯源字段；FAISS 映射是否可反查 |
| Agent 工具返回 | 五类工具是否统一返回结构（参见五中工具结果格式） |
| 后端↔前端↔Benchmark | `/api/qa` 的请求/响应字段三方是否一致（参见五中间答格式） |

**判定标准**：
- **符合**：上下游字段可对应（命名可不同，语义一致）
- **部分符合**：1-2 处不一致但可简单映射兼容
- **不符合**：多处字段断裂，下游引用的字段在上游产出中不存在

### 6.5 业务状态与关键参数管理

检查影响业务行为的参数是否可追溯、可调整：

| 检查项 | 说明 |
|--------|------|
| 业务阈值集中管理 | 如"短文本"字符数阈值、分块长度、Top-K 检索数量、相似度过滤阈值等是否集中在配置文件或模块常量中，非散落各处硬编码 |
| 数据状态流转 | `data_status` 的状态转换是否有明确规则（如 `valid` 的数据不会被后续流程改为 `empty_content`） |
| 文件/对象路径约定 | S3 bucket 名、数据目录路径、索引文件路径等是否在配置中统一约定，上下游一致引用 |
| 模型与参数绑定 | 向量模型名称、LLM 模型名称、temperature 等参数是否集中配置，确保问答和索引构建使用同一向量模型 |

**判定标准**：
- **符合**：关键业务参数集中管理，数据状态流转规则清晰，模型参数统一配置
- **部分符合**：部分参数集中但仍有少量散落，或状态流转有潜在冲突
- **不符合**：业务阈值大量硬编码且分散在各模块中，模型参数不一致
