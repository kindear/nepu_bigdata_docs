请基于项目代码、目录结构、README、配置文件、测试脚本和样例数据，对项目进行工程化代码评审。

## 评审目标
1. 检查项目结构清晰度与模块职责合理性；
2. 验证课程核心功能链路是否有代码证据；
3. 检查数据从采集→清洗→解析→知识库→问答→Benchmark 的贯通性；
4. 检查字段、接口、配置、测试的一致性；
5. 检查业务逻辑实现质量（流程完整性、业务规则正确性、边界条件处理、模块间数据契约）；
6. 给出可执行的修改建议。

## 重要原则
1. 不要求目录名、接口名、函数名与参考完全一致，等价实现且职责清楚即可；
2. 没有代码/配置/数据/测试/日志/README 证据的功能，不应默认完成；
3. 评审聚焦工程结构、模块边界、数据贯通、可运行性和业务逻辑实现质量（流程正确性、规则落实、边界覆盖）；不评价界面美观，不做安全专项，不逐项检查日志格式、类型标注、代码风格等辅助性规范；
4. 对学生自定义实现给出改进空间，不因命名不同直接判为不符合；
5. **前端代码不纳入详细检查**——仅确认前端目录存在且能对接后端即可，不对前端代码做逐项评审。

> 表格中「必选」列：✓ = 硬性要求，不满足即为 P0/P1；— = 加分项，不满足不扣分。

## 六、业务逻辑实现检查（4 项）

每项给出「符合 / 部分符合 / 不符合」判定，并附代表性代码位置（文件路径 + 行号或函数名）。

### 6.1 业务流程完整性

检查端到端数据处理链路是否可贯通执行，各阶段输入输出是否衔接：

| # | 检查项 | 说明 | 查证方式 |
|----|--------|------|----------|
| L1 | 采集→存储 | 采集结果可直接被存储模块消费（字段一致，非人工中转） | 比对采集输出字段与存储写入字段 |
| L2 | 存储→清洗 | PySpark 从数据库或 JSONL 读取采集结果（非手动导出 CSV 喂入） | 检查 PySpark 脚本的数据源（`spark.read.xxx`） |
| L3 | 清洗→解析 | 解析只处理 `data_status="valid"`；解析模块能读取 S3/本地文件 | 检查解析模块的输入过滤条件和文件读取方式 |
| L4 | 解析→知识库 | 解析产出的 `units` 可直接输入分块与向量化流程 | 检查 units schema 与分块输入是否匹配 |
| L5 | 知识库→问答 | FAISS 索引与映射文件被 RAG 检索模块加载并正常使用 | 检查问答模块是否 import FAISS 索引路径并调用 `index.search()` |
| L6 | 问答→Benchmark | Benchmark 调用 `/api/qa` 接口（非直接 import 调用内部函数绕过服务层） | 检查 Benchmark 是否 import 了 backend 内部模块——如有为 P0 |

**判定标准**：6 条均可找到衔接证据→**符合** / 1-2 处断层→**部分符合** / 3 处以上断层→**不符合**

### 6.2 业务规则正确性

对照三中各模块的功能要求，检查代码中的业务决策逻辑是否落实到位：

| # | 检查项 | 要求摘要（内联，无需回跳三） | 查证方式 |
|----|--------|---------------------------|----------|
| R1 | 去重规则 | 同时存在 URL 去重（`normalized_url`）和内容去重（`content_hash` 基于正文 MD5） | grep `normalized_url` + `content_hash` |
| R2 | 质量标记规则 | `data_status` 四类枚举值（valid/empty_content/short_content/invalid_time）判断分支完整 | grep `data_status` + 四类值的 `when`/`if` 分支 |
| R3 | 分块规则 | 块长度约 500 字、重叠 30-50 字；`chunk_id` 格式可追溯到源 document | 检查 `chunk_size`/`overlap` 参数；检查 chunk_id 含 document_id |
| R4 | 向量检索规则 | `IndexFlatIP` ↔ `normalize_embeddings=True` 必须配对 | 比对索引类型与归一化参数是否一致 |
| R5 | Agent 路由规则 | 存在问题类型判断（知识问答/网页查询/统计查询/附件查询），各分支可达 | 查看路由逻辑的 if-else 或 function call 分支覆盖 |
| R6 | Benchmark 独立会话 | 每条测试使用独立 `session_id`（如 `benchmark_test_001`），不共享聊天历史 | 检查 session_id 是否在循环体内生成新值 |

**判定标准**：6 条均实现且方向正确→**符合** / 1-2 条未实现→**部分符合** / 3 条以上缺失→**不符合**

### 6.3 边界条件与异常路径

检查业务异常场景是否有代码层面的处理逻辑：

| # | 检查项 | 要求摘要（内联） | 查证方式 |
|----|--------|----------------|----------|
| E1 | 空内容处理 | 空正文标记（非丢弃）；下游跳过空内容 | grep 空内容判断 + `data_status = "empty_content"` |
| E2 | 解析失败处理 | 单文件失败不阻断整批；失败原因可追溯 | grep `try`/`except` + 日志/错误记录（不应有裸 `continue`） |
| E3 | 检索无结果处理 | 返回"未找到"提示而非报错或编造 | 查找空结果分支（`if not results`） |
| E4 | 附件缺失处理 | S3 文件不存在时返回明确错误（非 500 崩溃） | 查找文件不存在的错误处理 |
| E5 | 超长输入处理 | 上下文超出 token 限制时有截断 | 查找 token 计数 + 截断逻辑 |
| E6 | Benchmark 失败保留 | 失败样例保留空记录（不跳过不丢弃） | 检查 try/except 中无 `continue` |

**判定标准**：≥4 种有明确代码处理逻辑→**符合** / 2-3 种→**部分符合** / ≤1 种→**不符合**

### 6.4 模块间数据契约

检查模块间传递的数据结构是否一致（关键字段定义参见四，接口格式参见五）：

| # | 检查项 | 查证方式 |
|----|--------|----------|
| C1 | 采集→存储→清洗：字段稳定 | 比对三阶段的 JSON/Parquet schema，确认字段名一致或可映射 |
| C2 | 解析→知识库：units 含溯源字段 | 检查 units 中含 `document_id`/`attachment_id`/`source_url` 等 |
| C3 | 分块→检索→问答：chunk_id + 映射可反查 | 检查 chunk JSONL 字段 + FAISS 映射文件 |
| C4 | Agent 工具返回：统一结构 | 检查五类工具的返回是否均为 `{success, data, message}` |
| C5 | 后端↔前端↔Benchmark：请求/响应一致 | 比对三方对 `/api/qa` 的调用代码和响应解析 |

**判定标准**：上下游字段可对应→**符合** / 1-2 处不一致但可映射→**部分符合** / 多处断裂→**不符合**
